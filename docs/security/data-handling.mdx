ZeroQuarry needs target data to perform security assessment. Treat each scan as
a controlled disclosure of the target material to the platform and the selected
LLM provider.

![ZeroQuarry account Git access page for managing private repository credentials.](/img/screenshots/account-git-access.png)

_Account-managed credentials such as private Git access are part of the same
data-handling story as scan inputs, logs, and provider exposure._

## Data used during scans

Depending on scan mode, ZeroQuarry may process:

- uploaded source files and archives
- cloned Git repository contents
- uploaded binary artifacts
- extracted and decompiled binary outputs
- remote HTTP responses
- scan notes and target metadata
- credentials or headers configured for remote probing
- finding chats and report context

## Logs

Scan logs show worker progress and tool activity. ZeroQuarry redacts sensitive
authentication header values from remote scan logs. Required bug bounty headers
are intentionally shown so operators can verify that each probe included them.

## Credentials

Saved Git credentials and account LLM keys are stored as account-level secrets.
Remote scan authentication values are stored with the scan context and attached
to probes as needed.

Sensitive credentials may still be visible to the LLM agent if they are required
for analysis or authenticated probing.

## Private-runner data boundary

Enterprise private runners execute eligible Git source and remote scans on a
customer-controlled Docker host. The runner makes outbound HTTPS connections to
ZeroQuarry and calls the selected LLM provider directly with account-managed
keys; no inbound firewall rule is required.

Each pool controls what returns to the SaaS control plane:

- **Minimized** returns allowlisted finding metadata and safe source locations.
  Free-form evidence, remediation text, logs, errors, and runner-generated
  artifacts remain on the runner.
- **Standard** also returns detailed findings, relevant excerpts, logs, and
  artifacts.

Minimized results reduce what returns to ZeroQuarry, but they are not a claim
that no data leaves the customer network: the selected LLM provider still
receives assessment context directly from the runner. Review both the pool
policy and provider configuration when defining the data boundary.

## Reports and artifacts

Reports, findings, exploit proofs, disclosure drafts, chats, and exports remain available
in the workspace until removed through the console or API. Deleting cloned
repositories for a scan removes local clone material while preserving findings
and report evidence.

## Recommended practices

- Use least-privilege credentials.
- Prefer staging targets for remote scans.
- Do not upload unrelated secrets or production data.
- Use private repository credentials scoped to the minimum required repos.
- Review generated exploit proofs and disclosure drafts before sharing them externally.
- Delete stale scan inputs and credentials when they are no longer needed.
